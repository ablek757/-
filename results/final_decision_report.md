# 智能客服意图识别：Prompt工程 vs 微调 — 完整评估报告

## 执行摘要

本报告对比了**Prompt工程**（4种策略）与**模型微调**（LoRA）在电商客服15类意图分类任务上的表现，从准确率、成本、延迟、易混淆意图等多维度进行分析，并给出产品决策建议。

**核心结论**：
- **Few-shot Prompt (v3)** 效果最佳（90.0%准确率），成本低，适合快速迭代
- **微调模型** 效果相当（89.5%准确率），延迟更低，适合生产环境
- **CoT推理** 适得其反（81.5%准确率），过度推理导致误判

**混淆原因分析**：

1. **订单修改 vs 退换货**：用户表述中同时包含"不想要"和"改地址"等模糊信号
2. **投诉建议 vs 售后维修**：负面情绪表达容易被误判为投诉，实际是维修需求
3. **价格优惠 vs 活动规则**：询问"能便宜吗"与"满减怎么用"边界模糊
4. **物流查询 vs 配送服务**："什么时候到"可能是查询进度或修改配送时间

**改进建议**：
- 在Few-shot示例中增加边界case样本
- 微调时对易混淆类别进行数据增强
- 考虑引入二级确认机制处理模糊query

## 5. 产品决策建议

### 5.1 什么场景用Prompt够了

✅ **推荐使用Few-shot Prompt的场景**：

1. **快速验证MVP**：需要快速上线验证产品假设
2. **意图体系频繁变化**：业务初期，分类标签还在调整
3. **小规模流量**：每日<1000次请求，成本敏感
4. **多租户SaaS**：每个客户意图不同，无法统一微调
5. **对延迟不敏感**：800ms响应时间可接受（如工单系统）
6. **无GPU资源**：团队无微调能力或算力资源

**Prompt工程最佳实践**：
- 使用Few-shot策略，每类2-3个示例即可
- 避免CoT推理，直接分类效果更好
- 定期review误判case，更新示例库

### 5.2 什么场景必须微调

✅ **推荐使用微调模型的场景**：

1. **生产环境稳定运行**：意图体系已固化，不再频繁变动
2. **大规模流量**：每日>10000次请求，成本优势明显
3. **延迟敏感**：实时客服场景，要求<500ms响应
4. **私有化部署**：数据不能出域，需本地推理
5. **特定领域术语**：垂直行业黑话多，通用模型理解差
6. **成本优化**：长期运营，训练成本可摊销

**微调最佳实践**：
- 至少准备800+标注样本，保证各类别均衡
- 使用LoRA等参数高效微调，降低训练成本
- 定期用新数据增量微调，保持模型时效性

### 5.3 混合策略

💡 **推荐混合方案**：

- **主路径**：微调模型处理常见意图（占90%流量）
- **兜底路径**：Prompt处理长尾/新增意图
- **置信度阈值**：微调模型置信度<0.8时，回退到Few-shot Prompt
- **持续学习**：收集Prompt处理的case，定期重新微调

## 6. 总结

| 维度 | Few-shot Prompt | 微调模型 | 推荐场景 |
|------|----------------|----------|----------|
| 准确率 | 90.0% ⭐ | 89.5% | 相当 |
| 成本（小规模） | ¥0.004/次 ⭐ | ¥0.06/次 | Prompt胜 |
| 成本（大规模） | ¥0.004/次 | ¥0.0004/次 ⭐ | 微调胜 |
| 延迟 | 800ms | 300ms ⭐ | 微调胜 |
| 迭代速度 | 分钟级 ⭐ | 小时级 | Prompt胜 |
| 稳定性 | 中 | 高 ⭐ | 微调胜 |
| 技术门槛 | 低 ⭐ | 中 | Prompt胜 |

**最终建议**：

1. **MVP阶段**：用Few-shot Prompt快速验证，2-3天上线
2. **成长期**：流量>1000/天时，启动微调项目
3. **成熟期**：微调模型为主，Prompt兜底，混合策略最优
4. **避免CoT**：本任务中CoT推理适得其反，不推荐使用
